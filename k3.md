## 1. 模型结构

![Kimi-K3 结构总览](k3.avif)

### 1.1 逐层混合骨架
```
KimiK3ForCausalLM → language_model
  ├─ embed_tokens
  ├─ layers[~95]: KimiDecoderLayer
  │    Attention: layer∈kda_layers → KimiKDAAttention(线性)   else → KimiFullAttention(MLA)
  │    FFN:       layer≥first_k_dense_replace → KimiSparseMoeBlock   else → KimiMLP(dense, layer0)
  ├─ norm + [output_attn_res_*]
  └─ lm_head
```

### 1.2 KDA 线性注意力(71 层)
门控 delta-rule 线性注意力(细粒度门变体)。5 路进核 + 1 路输出门:
```
q/k/v_proj ─cat→ causal_conv1d(+silu) → q,k,v   (读写 conv_state)
f_a→f_b_proj → 门 g     b_proj → beta     g_proj → 输出门
```
递归核 `_run_kda`(外部 `fla.ops.kda`),每 token 维护 `S[dv,dk]` 关联记忆(= `ssm_state`,fp32):
```
q,k=L2norm ; alpha=f(g,A_log,dt_bias)逐通道遗忘 ; b=sigmoid(beta)
S = S*alpha            # ④ 遗忘
v_old = S@k ; e = v-v_old
S = S + b*outer(e,k)   # ⑦ delta:擦旧写新
o = S@q                # ⑧ 输出
```
cache 定长 O(1):`conv_state`(bf16 滑窗)+ `ssm_state`(fp32 记忆矩阵)。

### 1.3 MLA 全注意力(24 层,head_dim=192)
DeepSeek-MLA 低秩投影,但 `use_mla=False`,**V padding 到 192 存标准 paged-MHA cache**。MQA 式 k_rope;
RoPE 仅作用 64 维 qk_rope;**输出 sigmoid 门** `attn_out*=sigmoid(g_proj(x))`。head_dim=192 见 §4。

### 1.4 MoE(`KimiSparseMoeBlock`)
DeepSeek-V3 路由(sigmoid + correction bias + grouped top-k)+ 可选 **latent 压缩**(专家在 `Rh` 隐维算,
down/up 全专家共享)+ **共享专家**(N 个融合成一个胖 MLP)+ **SiTU 激活**。
latent 路径 **all-reduce 在 norm 之前**(commit e68fa86)。

### 1.5 Attn-Residual(块残差,每 12 层)
沿深度方向的动态残差:维护快照栈,块内累加、块边界存快照并重置;每层 attn/mlp 前对
`[全部快照 + 当前流]` 做逐 token softmax 加权检索(`_apply_attn_res`),出网络再检索一次。

### 1.6 SiTU 激活
```
out = beta·tanh(gate/beta)·sigmoid(gate) · linear_beta·tanh(up/linear_beta)   (beta=4, linear_beta=25)
```
tanh 软钳制的有界 SiLU;`silu` 顶替会污染大幅值输出(曾致掉点)。

## 3. ★★ 当前 work-around(纯拖性能,应先拆除)

这些是为"跑通 / 跑对"临时加的兜底,每条都在拖性能。**关键是分清两类根因,拆除方式完全不同:**

- **A 类 = aiter 在该容器没编译全**:多个 CK-backed JIT 模块编译失败(CK 的 `get_compiler_target()`
  对目标架构报错),被逐个 bypass 到 torch/triton。**正解是把 aiter 正确编出来,不是算法问题。**
- **B 类 = K3 特有的 kernel / 算法 bug**:fla / aiter attention 在特定 shape(head_dim=192、短序列)上出错。
  **正解是修 / 换 kernel。**

> 注:A 类的 JIT 编译失败最早在 bring-up 时观察到(CK target detection);这些 fallback 在 gfx950 的 run
> 脚本里也开着。上线 gfx950 前应逐个确认:哪些 CK 模块**其实能编出来**(直接拆兜底),哪些**确实要修 CK**。

### A 类:aiter 未编译全 → torch/triton 兜底(编好 aiter 即可拆)

| Work-around | 开关 | 缺失的 aiter 模块 / 根因 | 性能代价 | 拆除条件 |
|---|---|---|---|---|
| **MoE 路由 torch topk** | `AITER_USE_TORCH_TOPK=1` | `module_moe_asm` 缺失 → 路由 topk 走 `aiter.ops.topk.*_torch`(torch 参考实现),不触发 asm kernel | 高:每个 MoE 层路由全跑 torch | 编出 `module_moe_asm`,切回 asm `biased_grouped_topk` |
| **关自定义 all-reduce** | `AITER_DISABLE_CUSTOM_ALL_REDUCE=1` | `module_custom_all_reduce` JIT 编译失败;CustomAllreduce 被禁用却仍用于 signal-buffer 注册 → 崩(`'CustomAllreduce' object has no attribute '_pool'`)→ 回退 PyNccl/torch all-reduce | 中-高:tp=8 每层多次 AR,custom AR(IPC one/two-shot)通常比 RCCL/PyNccl 快 | 修 CK 编译,开回 custom all-reduce |
| **triton quant** | `AITER_USE_TRITON_QUANT=1` | `module_quant` 编译失败(在线 bf16→MXFP4 量化 `get_hip_quant` 需要)→ 改用 triton quant 实现 | 低-中:量化走 triton 而非 CK/asm | 编出 `module_quant`,切回 CK/asm quant |
| *(opus rmsnorm)* | `AITER_USE_OPUS_RMSNORM=1` | `module_rmsnorm_quant` 编译失败 → RMSNorm 走 opus 后端 | ~无(opus 是真 kernel,非慢兜底) | 可保留 |

### B 类:K3 kernel / 算法 bug → 强制慢路(需修 / 换 kernel)

| Work-around | 位置 | 是什么 / 为什么 | 性能代价 | 拆除条件 |
|---|---|---|---|---|
| **KDA 强制 recurrent** | `ATOM_KDA_FORCE_RECURRENT=1`(kimi_k3.py:769) | KDA 正常是 **prefill 用 `chunk_kda`(把序列切块、块内并行,快)、decode 用 `fused_recurrent_kda`(逐 token 串行)**。此开关让 **prefill 也走串行 recurrent**。根因:fla `chunk_kda` 在短 prompt(seq<chunk)出 NaN,且它的 `transpose_state_layout` 输出与 decode 端 `fused_recurrent_kda` reader 的布局对不上 → 首个 decode step 就 NaN。强制 recurrent 让 prefill/decode 状态布局一致。 | **极高**:71 个 KDA 层 prefill 失去按块并行,退化成沿序列逐 token 串行 | 修 fla `chunk_kda` 短序列 NaN + 布局一致,或换原生 KDA kernel |
| **MLA prefill torch SDPA** | `_prefill_sdpa`(kimi_k3.py:415) | **为什么不走 aiter**:aiter 的 paged/unified attention **prefill kernel 在 head_dim=192 的多序列 varlen batch 上直接 fault(崩)**。所以 prefill 逐序列跑 torch `F.scaled_dot_product_attention`,KV cache 另行 `_write_kv_cache` 写入供 decode 读。 | 中:24 个 MLA 层 prefill 走 torch,非融合、不进 paged kernel | 修 aiter unified_attention 的 192 varlen prefill,再切回 |
| **MLA FLASH 布局(读)** | kimi_k3.py:487 / attention_mha | **为什么走 flash**:aiter `unified_attention` 的 **5-D SHUFFLE 读在 head_dim=192(非 2 幂)~100% 读错**(64/128/256 正常)。所以 MLA decode 绑到 4-D **FLASH** 布局 `unified_attention(shuffled_kv_cache=False)` 才能读对。**是的,根因就是 SHUFFLE 在 192 不支持/读错**。 | 中:decode KV 布局非最优(SHUFFLE 通常更省带宽) | 修 aiter 192 的 5-D SHUFFLE 读,评估切回 SHUFFLE |
| **Torch KV cache 写** | `ATOM_USE_TORCH_CACHE=1`(attention_mha.py:391) | 承上:选了 FLASH 布局后,写也要匹配 FLASH。aiter 本有 `reshape_and_cache_flash`,但此开关改用 torch `_torch_reshape_and_cache`(FLASH 写的额外兜底) | 中:每层 KV 写走 torch | 关掉,走 aiter `reshape_and_cache_flash` |
| **MoE SiTU torch fp32** | fused_moe_triton.py | triton MoE 的 `fused_clamp_act_mul` 只实现 silu/gelu,**无 situ** → 两个专家 GEMM 之间用 torch fp32 现算 situ(中间结果落 HBM 再读回) | 高:92 层不融合、多一次 HBM 往返 | 加 triton in-kernel situ(见 §4 P0-1) |
| **Torch sampler** | `ATOM_USE_TORCH_SAMPLER=1`(sampler.py:135) | 调试兜底;aiter fused topk/topp sampler 可用却被关 | 中:每 step 采样走 torch | 开 aiter fused topk/topp sampler |
| **关自定义 all-gather** | `ATOM_USE_CUSTOM_ALL_GATHER=0` | 与 custom all-reduce 一并保守关闭 | 中:TP/MoE all-gather | 验证后开回 |

> **别误拆(gfx950 预期快路)**:`ATOM_USE_TRITON_GEMM=1`、`AITER_USE_GROUPED_GEMM=1`、`AITER_USE_OPUS_RMSNORM=1`。
> `ATOM_FORCE_ATTN_TRITON=1`/`ATOM_USE_UNIFIED_ATTN=1` 为支持 192 强制 triton attn,修好 192 后再评估。
> **启动期兜底(只影响加载时间,不碰推理吞吐)**:`ATOM_LOADER_USE_THREADPOOL=0`、
> `ATOM_FASTSAFETENSORS_DIST_LOAD=0`、`ATOM_FASTSAFETENSORS_NOGDS=1`。

**拆除优先级**:① 先修 A 类(把 aiter 编好,一次性去掉 topk / quant / all-reduce 三个兜底)→
② B 类里收益最大的 **KDA 强制 recurrent**(prefill)与 **MoE SiTU 融合** → ③ **MLA 192**(prefill fault + SHUFFLE 读)→
④ sampler / KV cache / all-gather。**每拆一个复测一次 gsm8k(现 0.9378),确保精度不回退。**

## 4. ★ gfx950 性能重点(还要做什么)

> §3 是「拆掉现有兜底」,本节是「拆兜底所需的 kernel 工作 + 进一步融合」,两者互补(部分条目对应)。

按收益排序。核心矛盾:**gfx1250 靠 flydsl 融合 kernel 解决的东西,gfx950 都还没有,现在靠 triton + torch 兜底撑着。**

### P0 — MoE 相关(92 层,decode 主要开销)

1. **★★★ 融合 SiTU 进 triton MoE kernel**
   现状:MoE 专家两个 GEMM 之间,situ 用 **torch fp32** 现算(`raw_intermediate` 落 HBM fp32、再读回)。
   92 个 MoE 层每层每 token 一次,是 gfx950 MoE 的**最大浪费**。
   要做:在 aiter triton MoE(`moe_op_gemm_a16w4` / `fused_clamp_act_mul`)里加 **in-kernel situ**
   (参考公式 `beta·tanh(g/beta)·sigmoid(g)·lbeta·tanh(up/lbeta)`,beta/lbeta 传入或编译期常量),
   把 stage1→act→stage2 融成不落地。**这是 gfx950 MoE 的头号性能项。**

2. **★★★ 拆掉 MoE 路由的 torch 兜底**
   现状:`AITER_USE_TORCH_TOPK=1`,路由 topk 跑在 torch(调试兜底,慢)。
   要做:让 aiter 的 `biased_grouped_topk`(sigmoid+bias+grouped,asm)在 gfx950 上跑对,去掉 torch 兜底。
   每个 MoE 层都过路由,收益覆盖全模型。

3. **★★ MoE 激活量化 a16w4 → a8w4 / a4w4**
   现状:gfx950 用 `moe_gemm_a16w4`(**bf16 激活**),没吃到低精度激活的带宽/算力收益。
   要做:启用 a8w4/a4w4 激活量化的 MoE + **配套 situ**(注意 a8w4 GGUU decode 目前 silu-only、assert 拒绝 situ,
   需要加 situ 版才能开这条 decode 快路)。

### P1 — KDA / MLA(95 层的注意力主体)

4. **★★ KDA 在 gfx950 上原生化 / 调优**
   现状:71/95 层的 KDA 递归核 100% 靠**外部 fla**(triton),在 gfx950 上未必调优,且曾被列为数值可疑点。
   要做:评估 fla 在 gfx950 的性能;若差,做 aiter/自研 KDA kernel(chunk + recurrent),
   并把 **causal_conv1d 融进 KDA 前处理**(现在是独立 atom triton kernel)。这是单块最大的连续计算。

5. **★ MLA prefill 从 torch SDPA 迁到 aiter**
   现状:MLA prefill 走 `_prefill_sdpa`(torch),只有 decode 走 aiter unified_attention。
   要做:修好 unified_attention 的 192 varlen prefill fault 后,prefill 也走 aiter,省一条 torch 路径。

6. **★ MLA 192:FLASH vs SHUFFLE 布局**
   现状:因 aiter `unified_attention(shuffled_kv_cache=True)` 在 head_dim=192 误读,gfx950 被迫用 FLASH 4-D 布局。
   要做:修 aiter 的 192 SHUFFLE 5-D 读取;若 SHUFFLE 布局 decode 更快/更省带宽,可切回。

### P2 — 投影融合 / 小算子融合

7. **★ KDA in-proj 融合**:`q/k/v_proj`(+`b_proj`/`g_proj`)现为多个独立 GEMM;
   参考 Qwen GDN 的 `in_proj_qkvz(ba)` 融成一把大 GEMM,减少 kernel launch 与量化重复。

8. **★ MLA a-proj 融合**:`q_a_proj` 与 `kv_a_proj_with_mqa` 都从 hidden 出发,可融;`q_b`/`kv_b` 同理。

9. **KDA o_norm 用融合 gated RMSNorm**:`KimiRMSNormGated` 现为 torch;aiter 有 `gated_rmsnorm*` 可替换。

10. **Dense/shared SiTU 融合激活**:`SituAndMul` 现为 torch;暴露一个 aiter/triton `situ_and_mul`(类比 `silu_and_mul`)。

11. **`_apply_attn_res` 融合**:每层 2 次、~95 层的 torch(RMS 归一+打分+softmax+加权和),可融成一个小 kernel。

12. **AllReduce+RMSNorm 融合覆盖**:确认残差归一点(尤其 latent MoE all-reduce 后 → 下层 input_layernorm)都走
    aiter `tensor_model_parallel_fused_allreduce_rmsnorm(_quant)`,减少 all-reduce/HBM 往返。
