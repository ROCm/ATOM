## 1. 模型结构

![Kimi-K3 结构总览](k3.avif)

### 1.1 逐层混合骨架
```
KimiK3ForCausalLM → language_model
  ├─ embed_tokens
  ├─ layers[~95]: KimiDecoderLayer
  │    Attention: layer∈kda_layers → KimiKDAAttention(线性)   else → KimiFullAttention(MLA)
  │    FFN:       layer≥first_k_dense_replace → KimiSparseMoeBlock   else → KimiMLP(dense, layer0)
  ├─ norm + [output_attn_res_*]
  └─ lm_head
```

### 1.2 KDA 线性注意力(71 层)
门控 delta-rule 线性注意力(细粒度门变体)。5 路进核 + 1 路输出门:
```
q/k/v_proj ─cat→ causal_conv1d(+silu) → q,k,v   (读写 conv_state)
f_a→f_b_proj → 门 g     b_proj → beta     g_proj → 输出门
```
递归核 `_run_kda`(外部 `fla.ops.kda`),每 token 维护 `S[dv,dk]` 关联记忆(= `ssm_state`,fp32):
```
q,k=L2norm ; alpha=f(g,A_log,dt_bias)逐通道遗忘 ; b=sigmoid(beta)
S = S*alpha            # ④ 遗忘
v_old = S@k ; e = v-v_old
S = S + b*outer(e,k)   # ⑦ delta:擦旧写新
o = S@q                # ⑧ 输出
```
cache 定长 O(1):`conv_state`(bf16 滑窗)+ `ssm_state`(fp32 记忆矩阵)。

### 1.3 MLA 全注意力(24 层,head_dim=192)
DeepSeek-MLA 低秩投影,但 `use_mla=False`,**V padding 到 192 存标准 paged-MHA cache**。MQA 式 k_rope;
RoPE 仅作用 64 维 qk_rope;**输出 sigmoid 门** `attn_out*=sigmoid(g_proj(x))`。head_dim=192 见 §4。

### 1.4 MoE(`KimiSparseMoeBlock`)
DeepSeek-V3 路由(sigmoid + correction bias + grouped top-k)+ 可选 **latent 压缩**(专家在 `Rh` 隐维算,
down/up 全专家共享)+ **共享专家**(N 个融合成一个胖 MLP)+ **SiTU 激活**。
latent 路径 **all-reduce 在 norm 之前**(commit e68fa86)。

### 1.5 Attn-Residual(块残差,每 12 层)
沿深度方向的动态残差:维护快照栈,块内累加、块边界存快照并重置;每层 attn/mlp 前对
`[全部快照 + 当前流]` 做逐 token softmax 加权检索(`_apply_attn_res`),出网络再检索一次。

### 1.6 SiTU 激活
```
out = beta·tanh(gate/beta)·sigmoid(gate) · linear_beta·tanh(up/linear_beta)   (beta=4, linear_beta=25)
```
tanh 软钳制的有界 SiLU;`silu` 顶替会污染大幅值输出(曾致掉点)。

## 3. ★★ 生产脚本里的 work-around(需要拆的)

以生产脚本 `run_k3_mi355_tp8_cudagraph.sh` 为准。下面是它当前**仍在生效**、拖性能、需要逐个拆的
work-around(env 里的 torch 兜底该脚本都没开,不在此列)。

| Work-around | 位置 | 是什么 / 为什么 | 性能代价 | 拆除条件 |
|---|---|---|---|---|
| **关 prefix caching + 关 chunked prefill** | 命令行 `--no-enable_prefix_caching` `--no-enable_chunked_prefill` | **为什么关**:MLA 用 torch SDPA 做 prefill,只能看到 in-batch token;若开 prefix cache / chunked prefill,被缓存或分块的前缀会被漏掉 → 结果错。所以两个吞吐特性都被迫关闭。 | **高**:失去 KV 复用与长 prompt 分块两个主要吞吐优化 | 先修 MLA prefill(见下),让 MLA 走正规 paged prefill 后再开 |
| **MLA prefill torch SDPA** | `_prefill_sdpa`(kimi_k3.py:415) | **为什么不走 aiter**:aiter paged/unified attention 的 **prefill kernel 在 head_dim=192 多序列 varlen batch 上直接 fault(崩)**。故 prefill 逐序列跑 torch `F.scaled_dot_product_attention`,KV 另行 `_write_kv_cache` 写入供 decode。 | 中:24 个 MLA 层 prefill 走 torch,不进 paged kernel;并牵连上一条 | 修 aiter unified_attention 的 192 varlen prefill |
| **MLA FLASH 布局(decode 读)** | kimi_k3.py:487 / attention_mha | **为什么走 flash**:aiter `unified_attention` 的 **5-D SHUFFLE 读在 head_dim=192(非 2 幂)~100% 读错**(64/128/256 正常)。故 MLA decode 绑 4-D **FLASH** 布局 `shuffled_kv_cache=False` 才读对。**根因就是 SHUFFLE 在 192 读错。** | 中:decode KV 布局非最优(SHUFFLE 通常更省带宽) | 修 aiter 192 的 5-D SHUFFLE 读,评估切回 |
| **路由 MoE 专家 SiTU torch fp32**(默认 triton path) | `triton_kernel_fused_experts`(fused_moe_triton.py:347) | **路由专家**(`activation=Situv2` 传给 `FusedMoE`,kimi_k3.py:236)走默认 triton path;其 `fused_clamp_act_mul` 只实现 silu/gelu,**无 situ** → 两个专家 GEMM 之间用 torch fp32 现算 situ(中间落 HBM 再读回)。**这是拿到 0.9378 的默认路径,92 个 MoE 层全中**(commit 91968c5)。 | **高**:92 层不融合、每层多一次 HBM 往返 | **已有现成 fix**:`ATOM_USE_TRITON_MOE=0` 切 flydsl **native situv2**(+9% decode,~0.955 spot-check),但 flydsl grouped MoE 现在大 batch 会 illegal memory access(见 §4 P0-1) |
| **小 batch 绕开 contiguous-M grouped-gemm**(MoE 专家 GEMM) | `AITER_GROUPED_CONTIGUOUS_TOKEN_THRESHOLD=512` | 用在 **aiter flydsl grouped MoE 专家 GEMM(a8w4)**。它按 token 数选调度器:`token_num > 阈值` 走 **contiguous-M(DeepGEMM 式,route-indexed + 预shuffle 权重)**,否则走 **masked/非连续 grouped**。contiguous-M 在**小 prefill batch(~96 token)会 fault**;默认阈值 16 连小 batch 也走它 → 崩。设 512 让 ≤512 token 走 masked 路避开。 | 中:小 batch prefill 用非最优的 masked grouped(放弃 contiguous-M 的效率) | 修 contiguous-M(route-indexed preshuffle)在小 batch 的 fault,再把阈值调回 |
| **decode 走 unified_attention,而非更快的 gluon PA** | `ATOM_USE_UNIFIED_ATTN=1` / `ATOM_FORCE_ATTN_TRITON=1` + `use_flash_layout`(kimi_k3) | **三条 attention 路的取舍**(见下方详解):asm/CK PA 只支持 head_dim=128;**gluon PA(`run_pa_decode_gluon`,较快的 triton decode)读 5-D SHUFFLE cache,正好踩 192 SHUFFLE 读 bug**;K3 为躲该 bug 绑 4-D FLASH 布局,而 FLASH 只有 `unified_attention` 支持 → decode 被钉在 unified_attention。 | 中:decode 用不了 gluon PA(通常比 unified_attention 快) | 修 aiter 192 SHUFFLE 读 → 去掉 FLASH 兜底 → decode 可切 gluon PA |

> **别误拆(gfx950 预期快路,不是 work-around)**:`ATOM_USE_TRITON_GEMM=1`、`AITER_USE_GROUPED_GEMM=1`。
> **只影响启动、不碰推理(非性能 WA,不用拆)**:
> - `ATOM_USE_FASTSAFETENSORS=0` + 线程池 + 页缓存预热(~3min vs ~30min,仅加载)。
> - `ATOM_SYNC_AFTER_LOAD=1`:加载后一次性 TP barrier(`model_runner.py:669`),等所有 rank 加载完再 warmup,
>   避免加载时序竞争(某 rank 提前发 collective 会 hang)。**只跑一次,零推理开销**;加载够快/均匀后可去掉,但无性能收益。
> **必须保留的正确性开关**:`-tp 8`(tp4 OOM)、`--no-enable_prefix_caching/chunked_prefill`(在 MLA prefill 修好前不能开)。
>
> 注:contiguous-M 那条 kernel 在 `aiter/ops/flydsl/grouped_moe_gfx1250.py`(文件名带 gfx1250,但生产脚本按
> MI355 部署——本仓库对 MI355 的 arch 命名 gfx950/gfx1250 有混用)。它是 **MoE 专家 GEMM 的 grouped 实现**;
> K3 具体命中 flydsl grouped(a8w4)还是 ATOM triton(`moe_gemm_a16w4`)取决于 dispatch,值得进一步确认。

**详解:decode attention 为什么走 unified_attention 而不是 gluon PA**
ATOM 有三条 decode attention 路(`attention_mha.py`):
1. **asm/CK paged attention**(`run_pa_fwd_asm`/`pa_decode_bf16_asm`)—— 只支持 **head_dim=128**。K3 MLA=192,`use_triton_attn = FORCE or sliding_window!=-1 or head_dim!=128` 恒为真 → **被排除**。
2. **gluon PA decode**(`run_pa_decode_gluon`)—— 通常是**较快的 triton paged decode**,但它读 **5-D SHUFFLE KV cache**(`num_blocks, num_kv_heads, _, block_size, _ = k_cache.shape`),正是 aiter 在 **head_dim=192 读错**的那个布局。
3. **triton `unified_attention`** —— 支持 4-D FLASH 布局 + 任意 head_dim。

分派逻辑(`paged_attention_triton`):`if ATOM_USE_UNIFIED_ATTN or use_flash_layout: unified_attention(shuffled_kv_cache = not use_flash_layout) else: gluon PA`。
K3 为躲 192 的 SHUFFLE 读 bug 绑了 **FLASH 4-D 布局(`use_flash_layout=True`)**,`if use_flash_layout` 直接短路到 unified_attention,**永远进不了 gluon 分支**;加上生产又设了 `ATOM_USE_UNIFIED_ATTN=1`,双重钉死。
→ 所以要用上 gluon PA,前提是**修好 aiter 192 的 5-D SHUFFLE 读**,K3 才能回到 SHUFFLE 布局、走 gluon PA decode。

**重要澄清:head_dim=192 是「名义上支持、运行时有 bug」,不是「不支持」。**
`unified_attention`(`aiter/ops/triton/attention/unified_attention.py`)**没有** assert 拒绝 192;标准 Triton 2d kernel
用 **`HEAD_SIZE_PADDED`(必为 2 的幂,192→256)+ `if HEAD_SIZE_PADDED != HEAD_SIZE` 做 pad+mask** 来处理**任意** head_size,
代码里还专门为 *"Kimi-K3 non-power-of-two q head dim"* 把非 2 幂从 gluon 2d 路(`gl.arange(0,HEAD_SIZE)` 只能编 2 的幂)
引到标准 2d 路。所以 192 是**设计上支持**的。现存的是**两条具体路径在 192 上的运行时 bug**(都可修,不是缺特性):
- **decode + `shuffled_kv_cache=True`(5-D SHUFFLE)**:非 2 幂 192 读出~100% 错(64/128/256 正常)。
- **多序列 varlen prefill**:head_dim=192 时 fault(崩)。

修掉这两个 bug,FLASH 兜底 + torch SDPA prefill 都能拆,并解锁 gluon PA / prefix caching / chunked prefill。

**这两个是独立的两条轴,别混(常见误解:以为「FLASH + aiter + 192」会挂)**:
- **轴 A|布局(prefill 写 / decode 读 共享)**:5-D SHUFFLE 在 192 读错 → 改用 **4-D FLASH** 布局。
  **`4-D FLASH + aiter unified_attention + 192` 在 decode 上工作正常**——这就是当前 decode 的路,不挂。
- **轴 B|prefill 计算 kernel**:aiter 的 **varlen 多序列 prefill** 路径在 192 fault(与布局无关;同样 FLASH 布局,
  decode 形状能跑、prefill 形状崩,差别在 code path)→ prefill 改用 torch SDPA(只算 in-batch Q/K/V,不读 paged cache)。
- **prefill 与 layout 的关系**只在「**写**」:prefill 必须用 `reshape_and_cache_flash` 把 K/V 写成 decode 要读的 FLASH 布局
  (`_write_kv_cache`)。prefill 的**计算**(torch SDPA)与 layout 无关。

**拆除优先级**:① **修 aiter 192**(unified prefill fault + SHUFFLE 读)—— 一举解锁 MLA 走 aiter prefill、
并能**开回 prefix caching / chunked prefill**(最大吞吐收益)→ ② **MoE SiTU 融合进 triton**(92 层)→
③ 修 preshuffle grouped-gemm 小 batch fault(去掉 threshold 兜底)→ ④ 修好 192 后评估 attn kernel 选择。
**每拆一个复测 gsm8k(现 0.9378),确保精度不回退。**

## 4. ★ gfx950 性能重点(还要做什么)

> §3 是「拆生产脚本里的 work-around」,本节是「拆这些 work-around 所需的 kernel 工作 + 进一步融合」。

按收益排序。核心矛盾:**gfx1250 靠 flydsl 融合 kernel 解决的东西(如 in-kernel SiTU),gfx950 还没有,
现在 MoE 的 situ 靠 triton 外的 torch fp32 现算撑着。**

### P0 — MoE 相关(92 层,decode 主要开销)

1. **★★★ MoE SiTU 融合(两条路,择一)**
   现状:默认 triton path 的路由专家两 GEMM 之间用 **torch fp32** 现算 situ(落 HBM 再读),92 层的大浪费。
   - **路 A(已存在,优先)**:`ATOM_USE_TRITON_MOE=0` 切 aiter **FlyDSL native situv2** kernel(in-kernel 融合),
     实测 **+9% decode**、~0.955 spot-check。**阻塞点**:flydsl grouped MoE 在大 warmup batch(~8192 token)会
     `illegal memory access` 崩(`grouped_moe_gfx1250.py::_maybe_grouped_gfx1250_a8w4_moe`)。→ **修这个崩,即可把 flydsl 设默认。**
   - **路 B(备选)**:给默认 triton path 的 `moe_op_gemm_a16w4`/`fused_clamp_act_mul` 加 **in-kernel situ**
     (`beta·tanh(g/beta)·sigmoid(g)·lbeta·tanh(up/lbeta)`),stage1→act→stage2 不落地。
   **这是 gfx950 MoE 的头号性能项。** 另外 dense MLP(layer0)+ shared experts 的 situ 是 `SituAndMul`(torch,始终),
   也可顺带做成融合激活算子(量小,次要)。

2. **★★ MoE 激活量化 a16w4 → a8w4 / a4w4**
   现状:gfx950 用 `moe_gemm_a16w4`(**bf16 激活**),没吃到低精度激活的带宽/算力收益。
   要做:启用 a8w4/a4w4 激活量化的 MoE + **配套 situ**(注意 a8w4 GGUU decode 目前 silu-only、assert 拒绝 situ,
   需要加 situ 版才能开这条 decode 快路)。

### P1 — KDA / MLA(95 层的注意力主体)

4. **★★ KDA 在 gfx950 上原生化 / 调优**
   现状:71/95 层的 KDA 递归核 100% 靠**外部 fla**(triton),在 gfx950 上未必调优,且曾被列为数值可疑点。
   要做:评估 fla 在 gfx950 的性能;若差,做 aiter/自研 KDA kernel(chunk + recurrent),
   并把 **causal_conv1d 融进 KDA 前处理**(现在是独立 atom triton kernel)。这是单块最大的连续计算。

5. **★ MLA prefill 从 torch SDPA 迁到 aiter**
   现状:MLA prefill 走 `_prefill_sdpa`(torch),只有 decode 走 aiter unified_attention。
   要做:修好 unified_attention 的 192 varlen prefill fault 后,prefill 也走 aiter,省一条 torch 路径。

6. **★ MLA 192:FLASH vs SHUFFLE 布局**
   现状:因 aiter `unified_attention(shuffled_kv_cache=True)` 在 head_dim=192 误读,gfx950 被迫用 FLASH 4-D 布局。
   要做:修 aiter 的 192 SHUFFLE 5-D 读取;若 SHUFFLE 布局 decode 更快/更省带宽,可切回。

### P2 — 投影融合 / 小算子融合

7. **★ KDA in-proj 融合**:`q/k/v_proj`(+`b_proj`/`g_proj`)现为多个独立 GEMM;
   参考 Qwen GDN 的 `in_proj_qkvz(ba)` 融成一把大 GEMM,减少 kernel launch 与量化重复。

8. **★ MLA a-proj 融合**:`q_a_proj` 与 `kv_a_proj_with_mqa` 都从 hidden 出发,可融;`q_b`/`kv_b` 同理。

9. **KDA o_norm 用融合 gated RMSNorm**:`KimiRMSNormGated` 现为 torch;aiter 有 `gated_rmsnorm*` 可替换。

10. **Dense/shared SiTU 融合激活**:`SituAndMul` 现为 torch;暴露一个 aiter/triton `situ_and_mul`(类比 `silu_and_mul`)。

11. **`_apply_attn_res` 融合**:每层 2 次、~95 层的 torch(RMS 归一+打分+softmax+加权和),可融成一个小 kernel。

12. **AllReduce+RMSNorm 融合覆盖**:确认残差归一点(尤其 latent MoE all-reduce 后 → 下层 input_layernorm)都走
    aiter `tensor_model_parallel_fused_allreduce_rmsnorm(_quant)`,减少 all-reduce/HBM 往返。
