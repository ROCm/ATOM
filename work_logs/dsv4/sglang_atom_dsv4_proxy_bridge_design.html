SGLang-ATOM DeepSeek V4 Proxy Bridge 设计

这是当前 SGLang plugin enable DeepSeek V4 的整体设计说明。目标是最大化复用 ATOM DeepSeek V4 model、attention、cache layout 和 kernels，同时让 SGLang 保留上层 serving、scheduling 和 token allocation。

文档入口

1. 目标与原则

目标

DeepSeek V4 尽量复用 ATOM native model、attention kernel、compressor/indexer、mHC/MoE 逻辑。

ATOM model ATOM kernels ATOM cache layout

SGLang 保留

SGLang 仍然负责 server API、request lifecycle、scheduler、batching、token slot allocation 和 `ForwardBatch`。

scheduler ForwardBatch sampling

避免混用

不再把 ATOM V4 model 接到 upstream SGLang `DeepseekV4HipRadixBackend`，避免两套 cache/metadata 契约冲突。

no DSV4Metadata no SGLang V4 kernel path
当前路线对标 ATOM-vLLM #1060 的方案 1：上层框架分配/调度，ATOM plugin 解释 raw KV arena 并绑定回 ATOM V4 views。 第一版优先 correctness，暂不支持 prefix/radix cache 和 graph capture。

2. 总体架构

图中黄色箭头表示关键 bridge 行为：raw KV arena 不按 SGLang 原生 DSV4 pool 解释，而是被 ATOM plugin 切片成 ATOM V4 attention 需要的 `unified_kv`、`swa_kv`、CSA/HCA 和 indexer/cache state views。

3. 职责边界

组件 当前职责 不负责
SGLang Scheduler 管理 request lifecycle、prefill/decode/mixed mode、batch 合并、采样流程。 不理解 ATOM V4 内部 SWA/CSA/HCA/indexer/state cache layout。
SGLang ForwardBatch 承载 `input_ids`、`positions`、`seq_lens`、`req_pool_indices`、`out_cache_loc` 等运行时信息。 不拥有 ATOM metadata；只作为 bridge 输入。
ATOMDeepseekV4BackendForSgl SGLang backend shim：提供 backend 对象、保存 `token_to_kv_pool` 和 `req_to_token_pool`。 不执行 attention kernel，不构造 SGLang `DSV4Metadata`。
deepseek_v4_bridge.py 安装 proxy pool、切 raw arena、绑定 ATOM cache views、构造 ATOM `AttentionMetaData`。 不管理 `ForwardBatch` 生命周期，不做 sampling。
ATOM DeepSeekV4Attention 执行 native V4 attention、SWA write、compressor/indexer、sparse attention kernels。 不读取 SGLang `DSV4Metadata`。

4. Proxy KV Layout

Proxy Pool

`ATOMDeepSeekV4ProxyKVPool` 继承 SGLang 的 `BaseSWAKVPool`，这样能复用 SGLang `SWATokenToKVPoolAllocator` 的 full/SWA slot allocation 和 `full_to_swa_index_mapping`。

SGLang allocator -> full token slots -> SWA token slots -> full_to_swa_index_mapping -> proxy raw_arena

ATOM Views

bridge 将 `raw_arena` 按每层 `compress_ratio` 切成 ATOM native V4 所需 views。

per layer: SWA prefix: [num_slots, window, head_dim] bf16 if CSA: CSA main: [num_blocks, 32, head_dim] bf16 CSA indexer: [num_blocks, 32, aligned_index_dim] fp8 if HCA: HCA main: [num_blocks, 1, head_dim] bf16
当前版本为了第一版 correctness，先不支持 SGLang KV relocation / prefix cache / radix cache。`move_kv_cache()` 会直接报错， launch 需要 `--disable-radix-cache`。

5. Metadata Bridge

`build_atom_v4_attention_metadata_from_sglang()` 将 SGLang `ForwardBatch` 翻译成 ATOM V4 attention 所需 metadata。 它的作用类似 ATOM native `DeepseekV4AttentionMetadataBuilder`，但输入来自 SGLang。

SGLang 输入 ATOM metadata 输出 用途
`forward_batch.positions` `positions`、prefill/decode index builders RoPE、SWA ring offset、CSA/HCA visibility。
`forward_batch.seq_lens` / `extend_seq_lens` `cu_seqlens_q`、`context_lens`、`max_seqlen_q` 区分 decode、prefill、current chunk token count。
`req_pool_indices` + `req_to_token_pool` `block_tables` 给 compressor/indexer 计算 paged compressed KV offsets。
`out_cache_loc` / block id `state_slot_mapping` 给 per-request SWA ring 和 compressor state 分配稳定 slot。
`seq_lens // ratio` `n_committed_csa/hca_per_seq` 控制 C4/HCA compressed history 可见长度。

当前 debug 开关

ATOM_SGLANG_V4_DEBUG=1 打印 mode / total / positions / seq / state_slots ATOM_SGLANG_V4_DISABLE_COMPRESS_READ=1 临时把 n_committed_csa/hca 置 0，隔离 C4/HCA compressed-history read ATOM_V4_PREFILL_REF=1 使用 paged prefill torch reference attention，用于 kernel bisect

6. Forward Path

V4 attention patch

`patch_deepseek_v4_attention_for_sglang()` 不再把 ATOM attention 路由到 SGLang backend。它只做一件事： 当 SGLang 给 prefill 传 padded tensor，而 ATOM metadata 只覆盖 real token 时，slice real prefix 运行 native attention，再 pad 回原 shape。

7. 当前 Debug 结论

实验 结果 结论
绕过 SGLang LogitsProcessor，直接 `compute_logits()` 输出仍异常 问题不在 SGLang logits adapter 单点。
检查 padded prefill metadata 显示 `EXTEND total=4/5/6`，positions 同步 `#new-token=256` 是 SGLang bucket/统计口径，不是 ATOM 真实 token 数。
禁用 C4/HCA compressed read 输出仍不健康 问题不只是 compressed-history read。
ATOM-vLLM baseline 当前机器缺 vLLM image/package 和 #1060 bridge 文件 需要准备 vLLM 环境后再做 oracle 对比。
修复 indexer compressor `cache_scale` head_dim 已修 这是明确 bug，但不是当前短 prompt 首 token 异常的唯一原因。
当前现象：服务能跑，metadata/cache view 能绑定，但短 prompt 输出仍不正确。因为禁用 compressed-read 后仍异常， 下一步更应关注 current-chunk prefill attention、ATOM/SGLang metadata 细节差异、权重/config 是否完全对齐。

8. 下一步计划

A. 建立 ATOM-vLLM oracle

拉取或准备 `rocm/atom-dev:vllm-latest`，同步 #1060 的 `deepseek_v4_bridge.py`，用相同 checkpoint/prompt/采样参数比首 token top-logprobs。

vllm serve /workspace/.../DeepSeek-V4-Flash \ --port 8001 \ --tensor-parallel-size 8 \ --tokenizer-mode deepseek_v4 \ --no-enable-prefix-caching

B. 逐层 dump

同 prompt 下 dump SGLang bridge 与 ATOM-vLLM/native 的 embedding、第一层 attention output、若干层 hidden norm、最终 logits top-k。

compare: embed output norm layer0.attn output norm layer0 block output norm final hidden top-k logits

建议优先级

  1. 准备 ATOM-vLLM baseline，建立 logits oracle。
  2. 若 vLLM 正常，对齐 vLLM bridge 的 metadata build 与 SGLang bridge。
  3. 若 vLLM 也异常，回到 ATOM model/config/weight loading 路径检查。
  4. 最终再恢复 compressed read、radix/prefix cache、graph capture。
