1. 先建立心智模型

Upstream SGLang 的 DeepSeek V4 不是“一个模型类 + 一个普通 paged attention backend”，而是一套围绕 compress_ratios 展开的系统：模型层决定当前 layer 的 attention 类型，backend 负责为每个 forward 生成 V4 专用 metadata，KV pool 负责承载 SWA/C4/C128/indexer/state 多种物理存储，kernel path 再根据平台和 batch mode 选择 FlashMLA、HIP FlashMLA 或 sparse prefill。

Layer 视角

每层读取 compress_ratios[layer_id]，值为 0、4 或 128。这决定该层只看 SWA，还是额外读取 C4/C128 压缩历史。

Batch 视角

ForwardBatch 提供 request、seq length、cache slot。DSV4 backend 将它翻译成 DSV4Metadata，模型 forward 后续只消费 metadata。

Cache 视角

full KV slot 只是 scheduler 的统一地址空间；SWA pool 是缩小窗口空间，C4/C128 是压缩空间，不能互相直接替代。

AMD/ROCm 实现补充

AMD 侧的心智模型可以再加一层：同一个 DSV4 抽象下面，kernel 选择会被 HIP backend 改写。 DeepseekV4HipRadixBackend 保留 upstream metadata/cache 结构，但最终 attention dispatch 进入 flash_mla_with_kvcache_entrypoint(..., backend=os.environ["SGLANG_HACK_FLASHMLA_BACKEND"])。

Report 概念到 SGLang 实现

DeepSeek V4 technical report 中的关键思想是：不要让所有层都保存并读取完整历史 KV，而是组合近邻窗口、压缩历史和稀疏检索。 SGLang 把这个思想落成 compress_ratio = 0 / 4 / 128 三类 layer path。
Technical report 概念 SGLang 对应实现 工程含义
近邻窗口注意力 SWA_WINDOW = 128，swa_kv_pool，swa_page_indices 所有 layer 都保留最近窗口，decode 时每个 query 至少读 SWA cache。
C4 压缩历史 compress_ratio == 4，c4_kv_pool，C4Indexer C4 不是全量读取压缩历史，而是先通过 indexer 选 top-k，再给 FlashMLA 作为 extra cache。
C128 高压缩历史 compress_ratio == 128，c128_kv_pool 保存更稀疏、更便宜的长历史，page size 派生为 page_size / 128。
在线/分块压缩状态 CompressStatePool，get_compress_state_ring_size() compressor 需要跨 forward 维护窗口状态，因此 cache 不只是最终 KV，还包括 rolling state。
高吞吐 serving DSV4Metadata + FlashMLA scheduler metadata 把动态 request 转成 kernel 友好的 page/index/length 张量，减少模型层临时逻辑。

Source: DeepSeek V4 technical report and local SGLang implementation. Report link: DeepSeek_V4.pdf

AMD/ROCm 实现补充

Report 描述的是算法结构，AMD 实现要额外处理硬件友好的 layout：FP8 packed KV、RoPE 部分 BF16 保留、 scale padding、以及 HIP kernel 对 page/index 对齐的要求。因此在 AMD 上不要只看 attention 公式， 还要追踪 DeepSeekV4SingleKVPool.create_buffer() 中的 bytes-per-token layout 断言。

1ForwardBatch req_pool_indices seq_lens / out_cache_loc forward_mode
2init_forward_metadata decode / prefill / verify 选择分支
3make_core_attn_metadata page_table swa_page_indices raw_out_loc
4compression metadata C4 indexer metadata C4/C128 compress plan

上面的流程图表示每个 forward 之前 metadata 从 SGLang scheduler 的 batch 信息流向 DSV4 backend。关键是： attention kernel 并不直接读 request 对象，而是读 backend 预先构造好的 DSV4Metadata。

看源码时可以把 init_forward_metadata() 当成 DSV4 attention 的“编译阶段”： 它把 request 动态状态编译成 kernel 可直接消费的 page/index/length 张量。

AMD/ROCm 实现补充

AMD backend 复用这套 DSV4Metadata 思路，但更依赖 graph/capture 友好的张量生命周期。 如果打开 SGLANG_PREP_IN_CUDA_GRAPH 或类似 graph 路径，raw metadata 会在 graph 内升级成 full metadata。 ROCm debug 初期建议先关闭 graph，确保 swa_page_indices、extra_indices 和 swa_out_cache_loc 在 eager 路径下正确，再考虑 capture。

KV Cache 分层

SWA Pool

所有 layer 都用；保存最近窗口。full slot 需要映射到 SWA slot。

C4 KV + C4 Indexer Pool

C4 layer 使用。Indexer 先选 sparse positions，再读 compressed KV。

C128 KV Pool

C128 layer 使用。高压缩比历史，page size 派生为 page_size / 128。

Compress State Pools

跨 forward 维护 compressor rolling state，C4/C128 ring size 不同。

为什么不能用普通 KV Pool

DeepSeek V4 的 attention 不是 K cache + V cache 的单一结构。 它需要同时表达 SWA 窗口、C4 compressed history、C128 compressed history、C4 indexer cache 和 compressor state。

DeepSeekV4TokenToKVPool ├── swa_kv_pool ├── c4_kv_pool ├── c128_kv_pool ├── c4_indexer_kv_pool ├── compress_state_pools ├── indexer_compress_state_pools └── full_to_swa_index_mapping

因此 forward_batch.out_cache_loc 只是 full KV slot。 写入 SWA cache 时必须先经 translate_loc_from_full_to_swa() 得到 swa_out_cache_loc。

Pool sizing 逻辑

DSV4PoolConfigurator 会根据可用显存计算一个 full token budget，然后按 V4 attention 的压缩结构拆分成多类 pool。 这一步发生在模型实际 forward 之前，因此 launch 参数会直接影响后续所有 cache layout。

Pool 近似大小 配置/常量 说明
SWA pool full_token * swa_full_tokens_ratio --swa-full-tokens-ratio 只保存滑窗部分，不需要和 full token budget 等大。
C4 pool full_token / 4 compress_ratio == 4 HiSparse 打开时 device 侧 C4 pool 还可能按 host/device ratio 进一步缩小。
C128 pool full_token / 128 compress_ratio == 128 高压缩比历史，单位 token 成本低，但不能替代 C4/SWA。
C4 state pool swa_tokens / window * c4_ring_size get_compress_state_ring_size(4) 维护 C4 compressor 的 rolling state。
C128 state pool swa_tokens / window * c128_ring_size get_compress_state_ring_size(128) 维护 C128 compressor 的 rolling state；online compress 会改变 ring size 约束。

Full slot 与 SWA slot 的关系

forward_batch.out_cache_loc # scheduler 分配的 full KV slot -> token_to_kv_pool.translate_loc_from_full_to_swa(...) -> DSV4AttnMetadata.swa_out_cache_loc -> token_to_kv_pool.set_swa_key_buffer_radix_fused(...)

如果 debug 时看到 SWA cache 读写错位，优先检查 full_to_swa_index_mapping 是否已注册， 以及 swa_out_cache_loc shape 是否和当前 forward 的 out_cache_loc 对齐。

AMD/ROCm 实现补充

AMD 上的 KV cache layout 需要重点看 DeepSeekV4SingleKVPool： storage dtype 是按 FP8 packed layout 管理的，代码中断言默认 token layout 为 qk_nope_head_dim FP8(448) + qk_rope_head_dim BF16(64*2) + scale/pad(8)。 这会影响 fused store、FlashMLA 读取和 HiSparse/HICache 传输区域划分。

Attention Kernel Dispatch

模型层 prepare

入口是 MQALayer.forward()。它先完成 Q/KV projection、norm、RoPE 和 cache write。

MQALayer.forward -> _forward_prepare / _forward_prepare_multi_stream -> fused_q_norm_rope / fused_norm_rope_inplace -> SWA store -> C4/C128 compressor -> C4 indexer -> attn_backend.forward(..., save_kv_cache=False)

Backend kernel

入口是 DeepseekV4AttnBackend.forward() 或 HIP 版本。

DeepseekV4AttnBackend.forward -> get_swa_key_buffer_radix(layer_id) -> get_extra_key_buffer(layer_id) -> choose indices by compress_ratio -> flash_mla_with_kvcache -> flash_mla_sparse_fwd for large prefill

Kernel path 分解

阶段 典型函数/模块 触发条件 产物
Q prepare fused_q_norm_rope, fused_qk_norm_rope_swa_store 取决于 SGLANG_OPT_USE_FUSED_QK_NORM_ROPE、平台和 capture mode FlashMLA 可用的 Q tensor。
SWA KV write set_swa_key_buffer_radix_fused_norm_rope, set_swa_key_buffer_radix_fused 每个 layer 都需要；写入位置来自 swa_out_cache_loc SWA pool 中的 FP8/BF16 packed KV。

AMD/ROCm 实现补充

AMD kernel path 的核心分歧在 DeepseekV4HipRadixBackend.forward()： 它最终调用 flash_mla_with_kvcache_entrypoint，并用 SGLANG_HACK_FLASHMLA_BACKEND 选择 backend。若怀疑 kernel 正确性，先固定 backend=triton，再逐步切换到 kernel/AITER 路径比较结果。

关键约束

Page Size

page_size == 256 是当前 upstream V4 路径的硬约束。

C4 page size = 64；C128 page size = 2。

SWA Window

SGLang backend 中 SWA_WINDOW = 128。

SWA visible length 由 clamp(seq_lens, max=128) 得到。

Index Alignment

FlashMLA index last dimension 需要 64 对齐。

_pad_last_dim(..., multiples_of=64) 负责补齐。

AMD/ROCm 实现补充

AMD 上这些约束更像 correctness 前置条件，而不是性能建议。特别是 --page-size 256、 --kv-cache-dtype fp8_e4m3、indices 64 对齐和 SWA window 128。 任一项偏离，都可能不是“慢一点”，而是直接导致 HIP FlashMLA 或 fused store 读错 layout。

Decode、Prefill、Verify 的差异

Upstream SGLang 的 DSV4 backend 不是所有 forward 都走同一套 metadata 初始化。理解 forward mode 很重要， 因为它决定 seq_lens 如何展开、out_cache_loc shape 如何解释，以及 compressor metadata 如何构造。

Forward mode 入口函数 关键输入 主要差异
Decode / Idle init_forward_metadata_decode req_pool_indices, seq_lens, out_cache_loc 通常每个 request 一个新 token，metadata shape 接近 batch size。Idle 会走 dummy/padded 分支以保持图和张量形状稳定。
Prefill / Extend init_forward_metadata_prefill extend_seq_lens, seq_lens_cpu, out_cache_loc 一个 request 可展开成多个 query token，需要 expand_prefill_casually() 生成 token-level causal seq lens。
Target Verify init_forward_metadata_target_verify speculative draft token 数、verify batch 用于 MTP/spec decode 验证阶段，可能先生成 raw metadata，再在 graph 内升级成 full metadata。
Draft Extend init_forward_metadata_draft_extend draft token per batch 当前 V4 MTP top-k 有限制：代码中断言 MTP Topk > 1 not supported for DeepSeek V4。

为什么 prefill 更复杂

Decode 的每个 request 基本只新增一个位置；prefill/extend 会在一个 request 内新增一段 token。 因此 DSV4 backend 必须把 request-level 的 seq_lens 展开成 token-level 的 seq_lens_casual，这样每个 query token 都有正确的可见历史。

为什么 graph 内还有 raw metadata

CUDA graph / HIP graph replay 要求张量 shape 和内存位置稳定。SGLang 会在 out-of-graph 阶段保存 raw metadata， 再在 init_forward_metadata_in_graph() 里升级，确保 replay 时读取 live buffer。

AMD/ROCm 实现补充

ROCm 调试建议按 mode 分开验证：先 decode，再短 prefill，再长 prefill，最后 target verify/MTP。 HIP graph 和 multi-stream 会让 metadata 的准备位置从 Python eager 迁移到 graph 内或 stream overlap 中， 问题定位难度会明显上升。

ATOM Plugin 适配提醒

Upstream SGLang 的 DSV4Metadata 与 ATOM V4 的 AttentionMetaData_DSV4 不是同一套对象。 如果模型替换为 ATOM V4，就必须显式桥接 metadata/cache ownership；仅注册 dsv4 backend 不足以让 ATOM V4 attention 正确运行。

Upstream 原生路线

SGLang scheduler -> DeepSeekV4TokenToKVPool -> DSV4Metadata -> sglang.models.deepseek_v4 -> FlashMLA

ATOM plugin 路线

SGLang scheduler -> ForwardBatch / token_to_kv_pool -> bridge to ATOM AttentionMetaData_DSV4 -> atom.models.deepseek_v4 -> ATOM kernels

适配时最容易混淆的对象

Upstream SGLang V4 ATOM V4 为什么不能直接互换
DSV4Metadata / DSV4AttnMetadata AttentionMetaData_DSV4 字段集合和语义不同。SGLang 用 page/index/top-k metadata；ATOM 当前 V4 还需要 state_slot_mapping、compress_plans、block_tables 等。
DeepSeekV4TokenToKVPool ATOM per-request/unified V4 cache 设计 Upstream 拆 SWA/C4/C128/indexer/state pools；ATOM V4 attention 代码里有自己的 state/cache 绑定假设。
DeepseekV4HipRadixBackend atom.models.deepseek_v4.DeepseekV4Attention 前者是 SGLang backend；后者是 ATOM 模型内 attention。注册 backend 不会自动填充 ATOM forward context。
SGLang logits processor DeepseekV4ForCausalLM.compute_logits() V4 forward 返回 mHC residual stack，不是普通 [N, hidden]；不能直接套 lm_head 路径。

ATOM enable V4 推荐检查顺序

1. 模型注册：确认 SGLANG_EXTERNAL_MODEL_PACKAGE 命中 ATOM wrapper，而不是 upstream SGLang 原生模型。
2. Config 映射：确认 compress_ratios、window_size、index_topk、page_size=256 都保留下来。
3. Weight load：确认 V4 权重映射和 prefix="model." 对齐，特别是 compressor/indexer/head 权重。
4. Logits path：ATOM V4 必须调用 compute_logits()，不要走通用 lm_head。
5. Metadata bridge：把 SGLang ForwardBatch 和 cache 信息桥接成 ATOM AttentionMetaData_DSV4。
6. Cache ownership：明确是复用 upstream DeepSeekV4TokenToKVPool，还是让 ATOM 管理自己的 V4 cache。不要两边同时写。
7. Kernel alignment：固定 FP8 layout、page size、SWA window、C4/C128 page derivation，再逐步打开性能开关。

AMD/ROCm 实现补充

ATOM 在 AMD 上接 V4 时，最容易混合两套“AMD 实现”：upstream SGLang 的 HIP DSV4 backend 与 ATOM 自己的 V4 attention/kernel。前者围绕 DeepSeekV4TokenToKVPool + DSV4Metadata， 后者围绕 ATOM 的 AttentionMetaData_DSV4 和 ATOM kernel。二者都面向 AMD， 但 metadata/cache ownership 不同，不能因为都是 ROCm kernel 就直接拼接。
